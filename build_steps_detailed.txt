## 1. PROJECT VISION & ARCHITECTURE (THE BIG PICTURE)

### What is AcadAgent?
AcadAgent is an AI-powered academic advising assistant designed for BTech 
undergraduate students at IIT Gandhinagar (IITGN). It helps students do three things:
1. **Course Search & Logistics**: Look up credits, instructors, and class timings.
2. **Academic Advisor**: Generate optimal, step-by-step pathways to take courses 
   or pursue a minor discipline (e.g. Minor in Computer Science) based on prerequisite constraints.
3. **Policy & Syllabus Q&A**: Answer questions about course content, grading, 
   textbooks, and academic handbook rules (like credit limits, Honours, and Minors).

### The Architecture:
Instead of relying on a simple chatbot that guesses or makes up information, 
AcadAgent uses a **hybrid architecture** that coordinates:
* **SQLite Database**: A relational database that stores exact course timetables.
* **NetworkX Directed Graph**: A mathematical map of prerequisite relationships.
* **FAISS Vector Databases**: Search indexes that run semantic search over PDFs 
  (the Academic Handbook and Course Plans) using "embeddings."
* **LangGraph ReAct Agent**: The central AI engine that decides which of these 
  databases to query based on what the user asks.

## 2. PHASE 1: DATABASE INGESTION (EXCEL TO SQLITE)

### The Data Sources:
The raw data for course offerings, slot timetables, and instructors is located in 
two spreadsheets: `sem_odd.xlsx` (for the autumn semester) and `sem_even.xlsx` 
(for the spring semester).

### The Ingestion Code (`db/ingest_to_sqlite.py`):
1. **Reading Excel Files Safely**:
   * *Problem*: Standard Excel readers in Python (like Pandas) only read cell values 
     as plain text and throw away underlying links. This meant we would lose the 
     Google Drive/Doc links to official course plans.
   * *Solution*: We used the `openpyxl` library to read spreadsheets cell-by-cell. 
     By checking `cell.hyperlink.target`, we extracted the exact download links.
2. **Filtering for the Audience**:
   * Originally, we filtered out courses numbered 500 or higher (which represent 
     MTech/PhD courses). 
   * However, since BTech students can take advanced 500/600-level courses as 
     open electives (e.g., `ES 667` Deep Learning or `CS 613` NLP), we updated 
     the script to ingest **all** 625 courses so they are searchable.
3. **Semester Calculation**:
   * We wanted the database to tell the student what BTech semester a course is 
     recommended for.
   * We mapped this programmatically:
     * 100-level courses: Odd Semester = Semester 1, Even Semester = Semester 2.
     * 200-level courses: Odd Semester = Semester 3, Even Semester = Semester 4.
     * 300-level courses: Odd Semester = Semester 5, Even Semester = Semester 6.
     * 400-level courses: Odd Semester = Semester 7, Even Semester = Semester 8.
     * 500/600-level courses: Set to `None` since these are advanced electives.
4. **SQLite Insertion**:
   * We created a table named `courses` in a database file named `acadagent.db` 
     with columns: `course_code`, `clean_course_code` (standardized spacing like `CS 330`), 
     `title`, `instructor`, `semester` (odd/even), `semester_num` (BTech semester 1-8), 
     `L`, `T`, `P`, `C` (lecture-tutorial-practical-credits), `course_plan_url` (syllabus link), 
     `lecture_slots`, `tutorial_slots`, and `lab_slots`.

## 3. PHASE 2: RAG SEARCH ENGINES (PDFS TO VECTOR DATABASES)

### What is RAG?
RAG stands for **Retrieval-Augmented Generation**. Instead of training a model on 
your private documents (which is slow and expensive), you convert your documents 
into numbers called **embeddings** (representing the meaning of sentences) and 
store them in a vector database. When a user asks a question, the vector database 
finds the most relevant paragraphs from the document and feeds them to the LLM 
as "context" to write a factually accurate answer.

We built two RAG indexes:

### Index 1: The Academic Handbook (`db/build_vector_store.py`)
1. Read the `handbook.pdf` file.
2. Split the PDF text into smaller, overlapping chunks (e.g. 1000 characters 
   with a 200-character overlap) so that paragraphs aren't cut in half.
3. Sent these chunks to an embedding model (HuggingFace's local MiniLM model or 
   Gemini's embedding API) to convert the text into mathematical vectors.
4. Saved these vectors locally into a **FAISS (Facebook AI Similarity Search)** 
   index directory named `faiss_index_local`.

### Index 2: Course Plans (`db/build_course_plans_index.py`)
1. Checked SQLite for any courses that had a Google Drive syllabus URL.
2. **Bypassing Google Drive Warnings**:
   * *Problem*: When downloading files > 5MB, Google Drive displays a page warning 
     that "Google cannot scan this file for viruses," which breaks standard download scripts.
   * *Solution*: We wrote a Python session handler that grabs the virus scan 
     confirmation token from Google's response cookie and automatically resends the 
     request, downloading the PDF binary successfully.
3. Downloaded the PDFs to a folder named `scratch/downloaded_plans/`.
4. Read, chunked, and embedded the text of these syllabus plans into a FAISS index 
   named `faiss_index_course_plans_local`.

## 4. PHASE 3: PREREQUISITE GRAPH MAPPING (NETWORKX & TOPOLOGICAL SORT)

### What is a Directed Graph?
A directed graph is a mathematical network composed of **nodes** (points) and **edges** 
(directed arrows). In our project:
* Nodes represent courses (e.g. `ES 112`, `CS 201`, `CS 330`).
* Edges represent prerequisite relationships (e.g., an arrow pointing from `CS 201` 
  to `CS 330` means `CS 201` must be completed first).

### Building the Graph (`graph/build_prerequisite_graph.py`):
1. We set up a Pydantic schema `PrerequisiteRelation` requiring a `prerequisite` code, 
   `course` code, and the academic `rationale` (reason).
2. For each department prefix (`CS`, `EE`, `MA`, `ES`, etc.), we retrieved all 
   course codes and titles from SQLite.
3. Sent this list to Gemini with a structured prompt, asking the LLM to identify 
   factual prerequisite relationships and output them in structured JSON.
4. **Surviving Network Glitches**: We added a retry handler (3 attempts, 10s sleep) 
   so that temporary network glitches or 503 errors during this process would not skip 
   entire departments.
5. Saved the final graph structure (478 nodes, 119 edges) to `graph/prerequisite_graph.pkl` 
   using Python's `pickle` library.

### Prerequisite Resolution (How the Agent works out the path):
When a student asks for the sequence to take `CS 330`:
1. The agent loads the NetworkX graph.
2. It uses `nx.ancestors(G, target)` to recursively find all prerequisite nodes 
   (e.g., finding that `CS 330` requires `CS 201`, which requires `ES 112`).
3. It creates a small subgraph containing only these courses.
4. It runs a **Topological Sort** (`nx.topological_sort()`). This is an algorithm 
   that orders nodes in a linear sequence such that for every directed edge $U \rightarrow V$, 
   node $U$ comes before $V$.
5. This outputs a clean, sequential taking order (e.g., `ES 112` $\rightarrow$ `CS 201` $\rightarrow$ `CS 330`).

## 5. PHASE 4: AI BRAIN & REASONING LOOP (LANGGRAPH STATE MACHINE)

### What is LangGraph?
Instead of hardcoding a list of IF/ELSE statements to build the chatbot, we used 
**LangGraph**. LangGraph allows us to define the AI as a state machine:
1. **START**: Receives the student's text query.
2. **LLM Node**: The model processes the query. Based on what is requested, it 
   decides whether to output a final answer or call an academic tool.
3. **Tools Node**: Executes the requested tool (e.g. querying SQLite or FAISS) 
   and returns the outputs back to the LLM.
4. **Loop**: The LLM evaluates the tool outputs. If it needs more data, it calls 
   another tool. Otherwise, it moves to **END**.

### The Core Tools (`agent/tools.py`):
We equipped our agent with the following custom Python tools:
1. **`course_details`**: Takes a course code and fetches its title, instructor, credits 
   (L-T-P-C), recommended BTech semester number, slots, and syllabus links from SQLite.
2. **`course_schedule_details`**: Maps slot codes (like `A1`, `B2`, `H1`) to a dictionary 
   of exact days and times (e.g. Wednesday 10:00 - 11:20 AM) and prints a day-wise timetable.
3. **`prerequisite_finder`**: Finds direct prerequisites and rationales using the graph.
4. **`learning_path_generator`**: Performs the topological sort to output a sequential plan.
5. **`timetable_conflict_checker`**: Analyzes a list of courses and checks if any 
   share the same slot code in the same semester, flagging clashes.
6. **`policy_retriever`**: Performs semantic search over the academic handbook index.
7. **`course_info_retriever`**: Performs semantic search over the syllabus index.

### Smart Course Code Resolver:
* *The Problem*: Students often ask using shorthand or natural language (e.g. `"NLP"` or 
  `"what is operating systems"`). A direct database lookup for `"NLP"` would return empty.
* *The Solution*: We added `resolve_course_code` in `tools.py`. It uses a dictionary of 
  common shorthand abbreviations (like `NLP` $\rightarrow$ `Natural Language Processing`) 
  and checks SQLite using case-insensitive partial searches (`LIKE %term%`) to resolve 
  abbreviations or titles to their official codes (e.g. mapping `"NLP"` to `"CS 613"`).

## 6. PHASE 5: RESILIENCY & QUOTA MANAGEMENT (MODEL FALLBACK SYSTEM)

### The Quota Problem:
Gemini's free-tier API has strict rate limits (15 requests per minute, and a daily 
limit of 20 requests per model). During graph compilation or heavy testing, we 
frequently encountered `429 RESOURCE_EXHAUSTED` or temporary `503 Service Unavailable` errors.

### The Automated Fallback Solution (`agent/advising_agent.py`):
We implemented an **Automated Model Fallback Chain**:
1. We defined an ordered list of candidate models available on the API key:
   `MODELS_TO_TRY = ["gemini-2.5-flash-lite", "gemini-3.1-flash-lite", "gemini-2.0-flash", "gemini-2.5-flash"]`
2. In `run_advising_agent`, we loop through these models.
3. For the active model, we compile the LangGraph ReAct agent.
4. If the call fails with a 429 (Resource Exhausted) or 503 (Unavailable) error, 
   we immediately catch the exception, print a fallback log message, and switch to 
   the next model in the list.
5. This takes less than a second, isolating the user from quota limits.

## 7. PHASE 6: LOGGING, FRONTEND, AND UI INTEGRATION (STREAMLIT)

### The UI Layer (`app.py`):
We built a beautiful, modern user interface using **Streamlit**:
* **Theme Styling**: We injected custom HTML/CSS using `st.markdown(..., unsafe_allow_html=True)` 
  to build a dark mode layout with clean buttons, Google Fonts (Outfit), and orange accents.
* **Quick Queries**: We added 4 sample click buttons at the bottom so users can test 
  CS minor pathways, prerequisite orders, and timetable conflicts with a single click.

### The Streamlit Cache Trap (UI List Unpacking):
* *Problem*: LangChain returns messages containing lists of dictionaries for multimodal support. 
  Furthermore, Streamlit caches `st.session_state.messages` in the browser. When code was updated, 
  historical messages rendered as raw list brackets: `[{'type': 'text', 'text': ...}]`.
* *Solution*: We added a list unpacking loop in both the backend and the UI render loop:
  If a message content is a `list`, we loop through the parts, extract all `"text"` fields, 
  and merge them into a single string. We also added a **`🧹 Clear Chat History`** button to 
  instantly reset the browser cache.

### Detailed Execution Logger:
* Inside `advising_agent.py`, we added `save_interaction_log()`.
* Every query is appended to a local text file [interaction_logs.txt](interaction_logs.txt), 
  documenting exactly which models were tried, the fallback pathway, which tools were 
  called by the AI brain (with input arguments), the database responses, and the final answer.

## 8. PHASE 7: DEPLOYMENT & GITHUB LAUNCH

To make this project interview-ready, we initialized Git and published it:
1. **Adding `.gitignore`**:
   * We added a `.gitignore` file to ignore the virtual environment directory (`acad/` or `venv/`), 
     preventing thousands of binary files from bloating the commit.
   * We also ignored the `.env` file containing the secret Google API keys so they aren't 
     leaked to the public.
2. **Staging & Committing**:
   * Initialized Git locally: `git init`.
   * Staged all project files (timetables, database, scripts, graphs, and images): `git add .`.
   * Created the first commit: `git commit -m "Initial commit"`.
3. **Solving Git Permission Locks**:
   * *Problem*: The system had cached credentials for a global account (`xilinx-xhub-users-g1`), 
     denying permission to push to the personal repository (`AnshulKhamele/AcadAgent`).
   * *Solution*: We removed the cached credential from **Windows Credential Manager** under 
     `Generic Credentials -> git:https://github.com`. This forced Git to prompt for browser 
     authentication, allowing a successful push.

   * Created a detailed `README.md` containing an architecture diagram, visual graph references, 
     features overview, and execution instructions.
